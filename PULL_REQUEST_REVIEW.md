# Pull Request Reviews at Jahia

When you pick up a PR, start by actually reading the description — I know, sounds obvious, but it's easy to jump straight into the diff. Make sure the code does what the author says it does, poke at edge cases and error handling, and check it follows our conventions (functional style, strict TypeScript, tests that actually test something). If you're not sure whether it breaks things, run `npm run build && npm run lint && npm test` locally — it takes a minute and it's probably worth it. When leaving comments, explain *why* something should change, not just *what*; a naked "fix this" helps nobody. And don't forget: approve when it's good enough to ship, not when it's perfect — perfect doesn't exist, and we've all got other PRs in the queue.
